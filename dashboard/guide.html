The Agentic Workflow

Plan → Beads → Swarm → Review → Deploy

The Complete Development Lifecycle

A full development lifecycle encompassing ideation, planning, task breakdown, implementation, review, testing, deployment, and ongoing maintenance. Agents communicate via Agent Mail, manage tasks through a dependency-aware graph (Beads), and form a self-reinforcing ecosystem.

Phase 1: Ideation & Planning
Produce a comprehensive plan before any code is written
▼
Key principle: "It's a lot easier and faster to operate in 'plan space' before we start implementing." This phase takes ~1–2 hours. Output: a revised markdown plan (often 5,000+ lines).
1

Describe your project to your primary AI

Open ChatGPT (GPT-5.2 Pro), Claude Opus 4.5, or Gemini with Deep Think. Describe your project in detail — user experience, technical requirements, what you want to build. Ask it to create a comprehensive implementation plan.

2

Get competing plans from two other models

Give the same prompt to two other models. Each has different strengths:

  • GPT-5.2 Pro — system design and architecture
  • Claude Opus 4.5 — code quality and correctness
  • Gemini 3 with Deep Think — creative features and UX
3

Synthesize the best of all worlds

Paste all competing plans back into your primary AI with this prompt. Do 2–3 passes — it finds small oversights each time.

prompt · combine_plans 
I asked 3 competing LLMs to do the exact same thing and they came up with pretty different plans which you can read below. I want you to REALLY carefully analyze their plans with an open mind and be intellectually honest about what they did that's better than your plan. Then I want you to come up with the best possible revisions to your plan (you should simply update your existing document for your original plan with the revisions) that artfully and skillfully blends the "best of all worlds" to create a true, ultimate, superior hybrid version of the plan that best achieves our stated goals and will work the best in real-world practice to solve the problems we are facing and our overarching goals while ensuring the extreme success of the enterprise as best as possible; you should provide me with a complete series of git-diff style changes to your original plan to turn it into the new, enhanced, much longer and detailed plan that integrates the best of all the plans with every good idea included (you don't need to mention which ideas came from which models in the final revised enhanced plan):
4

Generate brilliant ideas (100 → top 10)

Run this 3+ times. The best ideas compound across passes.

prompt · 100_ideas 
OK so now I want you to come up with your top 10 most brilliant ideas for adding extremely powerful and cool functionality that will make this system far more compelling, useful, intuitive, versatile, powerful, robust, reliable, etc for the users. Use /effort max. But be pragmatic and don't think of features that will be extremely hard to implement or which aren't necessarily worth the additional complexity burden they would introduce. But I don't want you to just think of 10 ideas: I want you to seriously think hard and come up with one HUNDRED ideas and then only tell me your 10 VERY BEST and most brilliant, clever, and radically innovative and powerful ideas.
Then run: flywheel new "your idea" to start the local planning wizard — multi-model fan-out with synthesis happens automatically. Advance the gate to Beads when plan.md looks good.
2
Phase 2: Task Breakdown (Beads)
Convert the plan into a dependency-aware task graph
▼

Beads are granular tasks with explicit dependencies. Real examples: CASS project = 347 beads, SLB = 76 beads (14 epics, 62 tasks).

1

Create your project NTM session

ntm new myproject
2

Generate beads from your plan

In Claude Code with Opus 4.5, paste your final plan and send:

prompt · create_beads 
OK so please take ALL of that and elaborate on it more and then create a comprehensive and granular set of beads for all this with tasks, subtasks, and dependency structure overlaid, with detailed comments so that the whole thing is totally self-contained and self-documenting (including relevant background, reasoning/justification, considerations, etc.-- anything we'd want our "future self" to know about the goals and intentions and thought process and how it serves the over-arching goals of the project.) Use the `br` tool repeatedly to create the actual beads. Use /effort max.
3

Review and refine beads in "plan space"

prompt · improve_beads 
Check over each bead super carefully-- are you sure it makes sense? Is it optimal? Could we change anything to make the system work better for users? If so, revise the beads. It's a lot easier and faster to operate in "plan space" before we start implementing these things! Use /effort max.
4

Analyze and visualize with Beads Viewer

bv --robot-triage

Deterministic triage (recommended)

br ready

Show beads ready to work on

br stats

Project statistics overview

br blocked

Show blocked issues

bv --robot-insights

PageRank + critical path

bv --robot-plan

Parallel tracks, unblocks

Then advance gate: flywheel gate advance --next swarm

Phase 3: Agent Swarm Implementation
Multiple agents working in parallel via NTM + Agent Mail
▼
Real example: CASS project produced ~11k LOC in 5 hours, 204 commits, 151 tests passing.
1

Spawn agent sessions via NTM

flywheel swarm 6

Or directly: ntm spawn myproject 8

Run 3+ machines with multiple subscriptions: 5 Claude Max + 5 ChatGPT Pro + 3 Gemini Advanced. Scale as you see ROI.

2

Initialize each agent

Copy this to each agent pane to start them working:

prompt · new_agent 
First read ALL of the AGENTS.md file and README.md file super carefully and understand ALL of both! Then use your code investigation agent mode to fully understand the code, and technical architecture and purpose of the project. Then register with MCP Agent Mail and introduce yourself to the other agents. Be sure to check your agent mail and to promptly respond if needed to any messages; then proceed meticulously with your next assigned beads, working on the tasks systematically and meticulously and tracking your progress via beads and agent mail messages. Don't get stuck in "communication purgatory" where nothing is getting done; be proactive about starting tasks that need to be done, but inform your fellow agents via messages when you do so and mark beads appropriately. When you're not sure what to do next, use the bv tool mentioned in AGENTS.md to prioritize the best beads to work on next; pick the next one that you can usefully work on and get started. Make sure to acknowledge all communication requests from other agents and that you are aware of all active agents and their names. Use /effort max.
3

Agents self-coordinate via Beads + Agent Mail

br update ID --status=in_progress

Claim a bead before working

br close ID

Mark a bead complete

file_reservation_paths

Reserve files to avoid conflicts

send_message / fetch_inbox

Agent Mail communication

4

Safety: DCG + SLB

DCG blocks destructive commands before they run. SLB handles the "two-person rule" for high-risk operations, requiring quorum approvals via Agent Mail.

dcg doctor — verify the pre-commit hook is installed
Avoid "communication purgatory"! Agents should be proactive about claiming and completing tasks. If stuck waiting, move to other available beads. Continuous progress, not perfect coordination.

Phase 4: Review, Testing & Polish
Fresh reviews, peer reviews, UBS scan, UI polish, full tests
▼
1

Fresh self-review

prompt · fresh_review 
Great, now I want you to carefully read over all of the new code you just wrote and other existing code you just modified with "fresh eyes" looking super carefully for any obvious bugs, errors, problems, issues, confusion, etc. Carefully fix anything you uncover.
2

Peer review other agents' code

prompt · check_other_agents 
Ok can you now turn your attention to reviewing the code written by your fellow agents and checking for any issues, bugs, errors, problems, inefficiencies, security problems, reliability issues, etc. and carefully diagnose their underlying root causes using first-principle analysis and then fix or revise them if necessary? Don't restrict yourself to the latest commits, cast a wider net and go super deep! Use /effort max.
3

UI/UX scrutiny — Stripe-level polish

prompt · scrutinize_ui 
Great, now I want you to super carefully scrutinize every aspect of the application workflow and implementation and look for things that just seem sub-optimal or even wrong/mistaken to you, things that could very obviously be improved from a user-friendliness and intuitiveness standpoint, places where our UI/UX could be improved and polished to be slicker, more visually appealing, and more premium feeling and just ultra high quality, like Stripe-level apps.
4

Bug scan with UBS

prompt · apply_ubs 
Read about the ubs tool in AGENTS.md. Now run UBS and investigate and fix literally every single UBS issue once you determine (after reasoned consideration and close inspection) that it's legit.
5

Comprehensive tests

prompt · create_tests 
Do we have full unit test coverage without using mocks/fake stuff? What about complete e2e integration test scripts with great, detailed logging? If not, then create a comprehensive and granular set of beads for all this with tasks, subtasks, and dependency structure overlaid with detailed comments.
5
Phase 5: Deploy & Maintenance
Ship, then keep all projects moving every day
▼
1

Smart git commits (dedicated agent)

Use a dedicated agent to commit in logical groupings. Repeat every 15–20 min for multi-agent projects.

prompt · git_commit 
Now, based on your knowledge of the project, commit all changed files now in a series of logically connected groupings with super detailed commit messages for each and then push. Take your time to do it right. Don't edit the code at all. Don't commit obviously ephemeral files. Use /effort max.
2

Full GitHub deployment flow

prompt · gh_flow 
Do all the GitHub stuff: commit, deploy, create tag, bump version, release, monitor gh actions, compute checksums, etc.
3

Daily maintenance across ALL your projects

Make forward progress on every active project, every day — even when too busy for deep work. Use autopilot prompts (section below). Come back 3+ hours later to see incredible work done autonomously.

flywheel deploy — requires typing DEPLOY <project-name> to confirm
No human code edits post-agent work. Let agents handle git to avoid conflicts. Audit via exported mailboxes and bv time-travel features.

Daily Autopilot Mode
Keep every project moving with minimal effort
▼
The reality: Run these prompts on 8+ projects daily, keeping 3 machines busy constantly. Come back 3+ hours later to see incredible amounts of work done autonomously. The compound effect is incredible — models are good enough, and agents catch each other's mistakes.

Random code inspection

prompt · randomly_inspect 
I want you to sort of randomly explore the code files in this project, choosing code files to deeply investigate and understand and trace their functionality and execution flows through the related code files which they import or which they are imported by. Once you understand the purpose of the code in the larger context of the workflows, I want you to do a super careful, methodical, and critical check with "fresh eyes" to find any obvious bugs, problems, errors, issues, silly mistakes, etc. and then systematically and meticulously and intelligently correct them. Be sure to comply with ALL rules in AGENTS.md and ensure that any code you write or revise conforms to the best practice guides referenced in the AGENTS.md file.
